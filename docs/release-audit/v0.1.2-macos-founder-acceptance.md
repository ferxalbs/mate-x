## Before you start

### Artifacts (macOS RC only)

| Artifact | Use |
|----------|-----|
| `.dmg` | Preferred install path for founder testing |
| `.zip` | Alternate portable app bundle |
| `SHA-256` checksums | Verify integrity before install |

Install **only** from the RC artifact set for this SHA. Do not use `electron-forge start` or a dirty local `out/` tree for acceptance.

Record for every session:

- macOS version  
- Architecture (arm64 / x64)  
- Artifact filename + SHA-256  
- Source commit from the RC evidence / checksum companion  
- Date / time  

### Environment hygiene

1. Prefer a clean app data profile if prior development builds were used.  
2. Use a **real, non-sensitive** test repository you control.  
3. Rainy: test both **with** valid credentials and **without** (blocked path).  
4. Capture screenshots only for failures or intentional pass evidence.  
5. Do not silently fix product issues during acceptance — log and retest after focused fixes.

### Severity rubric

| Severity | Meaning | Merge to main? |
|----------|---------|----------------|
| **P0** | Security, data loss, app unusable | **Blocked** |
| **P1** | Primary workflow broken | **Blocked** |
| **P2** | Significant UX or reliability issue | Track; fix or explicit accept |
| **P3** | Polish | Track |

**Do not merge into `main` while P0 or P1 issues remain.**

Log findings in `docs/release-audit/v0.1.2-founder-results.md`.

## Founder scenarios (20)

### 1. Install the DMG

| | |
|--|--|
| **Action** | Open the RC `.dmg`, drag MaTE X to Applications (or open from the volume if that is the intended path). |
| **Expected** | Install completes without cryptic native-module or signature dialogs that block launch. |
| **Fail if** | Installer fails, app will not copy, or system blocks without a recoverable path. |

### 2. Launch MaTE X normally

| | |
|--|--|
| **Action** | Launch MaTE X from Applications / Launchpad / Dock like any desktop app. |
| **Expected** | Main window appears; product is usable without terminal or forge. |
| **Fail if** | Crash on launch, blank forever, or requires developer tooling. |

### 3. Confirm no ripgrep crash

| | |
|--|--|
| **Action** | Use features that search the repository (open repo, search, engineering analysis paths). |
| **Expected** | No crash or error dialog about missing `rg` / ripgrep / native binary. |
| **Fail if** | Process dies on search, or error surfaces a missing unpacked binary. |

### 4. Open a real test repository

| | |
|--|--|
| **Action** | Open a real local git repository you control (not a synthetic CI fixture). |
| **Expected** | Repo opens successfully; product treats it as a normal project. |
| **Fail if** | Open fails, hangs, or only works on MaTE X’s own repo. |

### 5. Confirm the real repository and branch

| | |
|--|--|
| **Action** | Check displayed project name and current branch. Optionally switch branch. |
| **Expected** | Name and branch match reality (not a hardcoded `main` or wrong path). |
| **Fail if** | Wrong branch, wrong name, or UI lies about trust/access. |

### 6. Submit an engineering objective

| | |
|--|--|
| **Action** | Submit a real engineering objective in plain language (raw user intent). |
| **Expected** | Task is captured; product begins a structured engineering journey. |
| **Fail if** | Objective is ignored, crashes, or immediately marked complete. |

### 7. Confirm the raw prompt is not treated as a completed specification

| | |
|--|--|
| **Action** | Observe status after capture / intent compilation. |
| **Expected** | Raw prompt is **not** frozen as an approved specification without human review. |
| **Fail if** | Product skips to “done” or treats free text as approved Spec. |

### 8. Review clarification questions

| | |
|--|--|
| **Action** | Answer any clarification questions the product raises. |
| **Expected** | Questions are relevant and improve the specification; answers stick. |
| **Fail if** | Nonsense questions, loop, or answers discarded. |

### 9. Review and approve specification

| | |
|--|--|
| **Action** | Review the compiled specification; approve when correct. |
| **Expected** | Explicit human approval gate; refusal/edit path exists. |
| **Fail if** | Spec auto-approves without consent, or approval is non-functional. |

### 10. Review and approve plan

| | |
|--|--|
| **Action** | Review the work plan / task graph; approve to proceed. |
| **Expected** | Plan is inspectable; approval is deliberate. |
| **Fail if** | Plan hidden, auto-approved, or approval does nothing. |

### 11. Execute through Rainy

| | |
|--|--|
| **Action** | Run execution with Rainy credentials when available; also try missing credentials. |
| **Expected** | With credentials: execution proceeds with structured progress. Without: clear blocked state, not a hang or fake success. |
| **Fail if** | Silent failure, fake “success”, or unusable credential error. |

### 12. Inspect structured progress

| | |
|--|--|
| **Action** | Watch execution progress while work runs. |
| **Expected** | Progress is structured and truthful (phases / steps), not a black box spinner only. |
| **Fail if** | No progress signal, or progress clearly false. |

### 13. Run validation

| | |
|--|--|
| **Action** | Trigger or observe validation after execution. |
| **Expected** | Validation runs and reports pass/fail with usable output. |
| **Fail if** | Validation skipped, always-green without work, or unusable. |

### 14. Inspect coverage convergence

| | |
|--|--|
| **Action** | Review coverage / convergence signaling for the task. |
| **Expected** | Product shows whether requirements/tasks are converging toward ready-to-ship. |
| **Fail if** | Coverage UI is empty, contradictory, or claims ship when incomplete. |

### 15. Inspect Ship Proof

| | |
|--|--|
| **Action** | Open Ship Proof for the task when available. |
| **Expected** | Proof is inspectable and tied to real validation / anchors — not decorative. |
| **Fail if** | Proof missing when claimed ready, or pure marketing text with no substance. |

### 16. Verify GitGate blocks stale or missing proof

| | |
|--|--|
| **Action** | Attempt a git-gated ship action without valid proof, or with stale proof (e.g. after local change). |
| **Expected** | GitGate **blocks**; reason is understandable. |
| **Fail if** | Ship proceeds with missing/stale proof, or block is silent/confusing. |

### 17. Quit and reopen MaTE X

| | |
|--|--|
| **Action** | Fully quit MaTE X, then relaunch. |
| **Expected** | Clean shutdown and restart without crash loops. |
| **Fail if** | Hang on quit, crash loop, or data wipe. |

### 18. Confirm EngineeringTask recovery

| | |
|--|--|
| **Action** | After reopen, locate the EngineeringTask from the session. |
| **Expected** | Task state recovers from durable storage; work is not lost. |
| **Fail if** | Task vanished, status reset incorrectly, or corruption. |

### 19. Evaluate speed and UX

| | |
|--|--|
| **Action** | Subjectively rate startup, window show, interaction, and task cycle feel. |
| **Expected** | Responsive enough for daily use; note rough timings if slow. |
| **Fail if** | Unusable latency (P0/P1) or chronic multi-second freezes (P1/P2). |

### 20. Confirm the product feels like MaTE X

| | |
|--|--|
| **Action** | Holistic judgment after the walkthrough. |
| **Expected** | Feels like **MaTE X** — trust layer for AI-written code — not a generic IDE clone or internal test dashboard. |
| **Fail if** | Visible test/audit harness UI, Factory identity, Spec Kit branding as product, or pure IDE packaging without MaTE X purpose. |

## Installation instructions (founder)

1. Obtain the macOS RC set for `v0.1.2-rc.1` (DMG + ZIP + SHA-256).  
2. Verify:

   ```bash
   shasum -a 256 -c <checksums-file>
   ```

3. Open the DMG and install MaTE X.  
4. Launch from Applications.  
5. Walk scenarios 1–20.  
6. Log every issue with severity P0–P3 in `v0.1.2-founder-results.md`.
