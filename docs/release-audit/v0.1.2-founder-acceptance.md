# Founder Acceptance Test Plan — MaTE X v0.1.2-rc.1

**Product:** MaTE X  
**Candidate:** `v0.1.2-rc.1`  
**Role of tester:** Founder — experience the product as a **user**, not as a test operator  
**Authority:** Agent 6 (Release Candidate Integrator). Does **not** authorize public release.  

**What this is:** A product walkthrough. Install the release candidate, do real engineering work, and decide whether the product earns trust.

**What this is not:** Inspection of internal contracts, ledgers, fixtures, CI logs, or implementation details — unless diagnosing a failure.

---

## Before you start

### Artifacts (use only the RC build)

| Platform | Install from | Verify integrity |
|----------|--------------|------------------|
| macOS | `.dmg` or `.zip` from RC set | Match SHA-256 in RC checksum file |
| Windows | `Setup.exe` (and required package files) | Match SHA-256 in RC checksum file |

Record for every session:

- Platform and OS version  
- Architecture (arm64 / x64)  
- Artifact filename + SHA-256  
- Source commit printed in the RC manifest (or RC evidence doc)  
- Date / time of the session  

### Environment hygiene

1. Install from the RC artifact only (not `electron-forge start`, not a local `out/` from a dirty tree).  
2. Use a **real, non-sensitive** test repository you control (public sample or private sandbox). Do not use production secrets.  
3. Prefer a clean app data profile if you have used prior development builds.  
4. Rainy credentials: test both **with** valid credentials and **without** (blocked path).  
5. Capture screenshots only when something looks wrong or for pass evidence you want on record.

### Pass / fail rule of thumb

| Result | Meaning |
|--------|---------|
| **Pass** | Behavior matches expected; product feels intentional |
| **Fail (P0–P1)** | Security, trust, or core flow broken — blocks merge |
| **Fail (P2–P3)** | Major UX / polish — track; P2 needs fix or explicit acceptance |
| **Blocked** | Environment issue (network, credentials, wrong artifact) — note and retest |

Log every issue in `docs/release-audit/v0.1.2-founder-results.md` using the structured template. **Do not silently fix** during acceptance; raise issues and retest after focused fixes.

## Scenario A — Installation and startup

**Goal:** The app installs and launches like a shipping desktop product.

| Step | Action | Expected |
|------|--------|----------|
| A1 | Install from RC artifact | Installer / drag-install completes without cryptic native errors |
| A2 | Launch MaTE X normally | Window appears; no crash on first open |
| A3 | Watch for dependency failures | No ripgrep / native module crash dialogs |
| A4 | Subjective startup | Feels responsive enough for daily use (note rough seconds if useful) |
| A5 | Scan chrome / menus / first screens | **No** test UI, audit UI, Factory product identity, Spec Kit branding, internal qualification panels, or harness controls |

**Fail if:** Crash on launch, missing native deps, or any visible “test / audit / Factory / Spec Kit” operator surface.

---

## Scenario B — Repository opening

**Goal:** MaTE X shows **truthful** repository context.

| Step | Action | Expected |
|------|--------|----------|
| B1 | Open a real non-sensitive test repository | Repo opens successfully |
| B2 | Check displayed name | Matches the actual folder / remote name |
| B3 | Check branch | Shows the **actual** current branch (not a hardcoded `main`) |
| B4 | Check permission / trust state | Reflects real trust/permission state — not a blanket “Full access” when that is false |
| B5 | Switch branch or open a second repo if easy | UI updates to the new truth |

**Fail if:** Hardcoded `main`, false “Full access”, wrong name, or silent wrong branch.

## Scenario C — Engineering objective capture

**Goal:** An objective becomes a real engineering task, not a completed “spec” by dumping the prompt.

**Suggested objective (paste as written or adapt slightly):**

> Add rate limiting to the public API while preserving existing authentication behavior and produce evidence that the change is safe to ship.

| Step | Action | Expected |
|------|--------|----------|
| C1 | Submit the objective | MaTE X creates **one** EngineeringTask (single task identity for this work) |
| C2 | Observe status / framing | Raw prompt is **not** treated as a finished specification |
| C3 | Review what MaTE X identifies | Scope, non-goals, requirements, and acceptance criteria appear (or clear path to them) |
| C4 | Clarifications | Asks only questions that **materially** change the work; no questionnaire spam |

**Fail if:** Prompt auto-marked complete as Spec; no task created; meaningless or endless clarification.

---

## Scenario D — Specification and plan

**Goal:** You understand and control the plan; the agent cannot approve itself.

| Step | Action | Expected |
|------|--------|----------|
| D1 | Read the specification | Understandable to a technical founder without reading source code |
| D2 | Check exclusions | Non-goals / exclusions are explicit |
| D3 | Check acceptance criteria | Testable (you can imagine pass/fail) |
| D4 | Read technical plan | Plan steps trace back to requirements |
| D5 | Approve or reject | You can do either; reject does not force execution |
| D6 | Trust boundary | Agent **cannot** self-approve the plan |

**Fail if:** Opaque plan; agent self-approves; no approve/reject control; criteria untestable.

---

## Scenario E — Execution

**Goal:** Scoped, visible, cancelable execution with honest failure states.

| Step | Action | Expected |
|------|--------|----------|
| E1 | Before write work | MaTE X explains what it is about to change |
| E2 | During execution | Work stays scoped to the opened repository |
| E3 | Task visibility | Current task / phase is visible |
| E4 | Cancel | Cancellation works and does not leave a “completed” lie |
| E5 | Failure path | Failure remains failure (not auto-completion) |
| E6 | Missing Rainy credentials | Clear **blocked** state with understandable next step |
| E7 | With valid credentials (if available) | Structured activity / progress, not a silent black box |

**Fail if:** Silent writes; work outside repo; cancel broken; failure marked success; missing credentials crash or pretend success.

---

## Scenario F — Validation and convergence

**Goal:** Evidence is real, visible, and blocks false readiness.

| Step | Action | Expected |
|------|--------|----------|
| F1 | Validation commands | Visible and relevant to the change |
| F2 | Failed validation | Blocks readiness / ship-ready claims |
| F3 | Coverage gaps | Gaps are understandable (what is unproven) |
| F4 | Stale evidence | Old evidence is not accepted after material change |
| F5 | Final proof | Connects requirements → changes → validation |

**Fail if:** Green readiness with failed/missing validation; opaque coverage; stale proof accepted.

---

## Scenario G — GitGate

**Goal:** Commit/push require fresh proof; no emergency bypass.

| Step | Action | Expected |
|------|--------|----------|
| G1 | Attempt commit/push without fresh proof | Blocked |
| G2 | Mutate the repo externally (e.g. edit a file / commit outside MaTE X) | Prior proof becomes invalid / gate re-blocks |
| G3 | Read the UI | Explains **why** the operation is blocked |
| G4 | Recover | Fresh validation is required to proceed |
| G5 | Bypass hunt | No emergency “force commit” / “bypass safety” control exposed |

**Fail if:** Unproven commit/push succeeds; silent invalidation; no explanation; bypass control present.

---

## Scenario H — Restart recovery

**Goal:** Incomplete work survives restart; approvals and evidence remain; stale repo state is detected.

| Step | Action | Expected |
|------|--------|----------|
| H1 | During an incomplete task, quit MaTE X fully | App exits cleanly |
| H2 | Reopen MaTE X | EngineeringTask is still there and resumes correctly |
| H3 | Approvals | Prior approvals still present if still valid |
| H4 | Evidence | Completed evidence still present if still valid |
| H5 | Stale repo | If the repo changed while closed, stale state is detected (not silently trusted) |

**Fail if:** Task lost; false clean slate; approvals/evidence vanish without cause; stale repo treated as fresh.

## Scenario I — Product identity

**Goal:** The experience matches the product promise.

**Must feel like:**

> MaTE X directs and verifies engineering work.

**Must not feel like:**

- an IDE  
- a generic chatbot  
- a collection of modes  
- Spec Kit  
- Factory Run  
- a test dashboard  
- an internal developer tool  

| Step | Action | Expected |
|------|--------|----------|
| I1 | After a full or partial cycle, write one sentence of your impression | Matches “directs and verifies engineering work” |
| I2 | Scan for legacy product names / mode-picker theater | Absent or clearly non-authoritative |
| I3 | Trust posture | Product prioritizes proof over chat fluency |

**Fail if:** Strong IDE/chatbot/mode/Factory/Spec Kit/test-dashboard identity.

## After the session

1. File every defect in `docs/release-audit/v0.1.2-founder-results.md`.  
2. P0/P1 → merge blocked.  
3. P2 → fix or explicit written acceptance.  
4. P3 → optional polish backlog.  
5. Agent 6 (or release coordinator) issues exactly one final gate string after your decision:

   - `READY TO MERGE PR #9 INTO MAIN — RELEASE NOT YET AUTHORIZED`  
   - or `DO NOT MERGE PR #9 — ACCEPTANCE BLOCKERS REMAIN`

**Still not authorized without separate founder approval:** merge PR #9, tag `v0.1.2`, publish artifacts, GitHub Release, public distribution.
