# MaTE X v0.1.2 Rollback Plan

## Rollback point

| Field | Value |
| --- | --- |
| Release branch (work) | `audit/v0.1.2-production-hardening` |
| Candidate commit | `aa2ed86827aad780b9e375ff4cf2e6de22d3261d` |
| Prior RC documentation commit | `6da6777ebe99f34e50cef70074e5bcf86c28c5bc` (`v0.1.2-rc.1`) |
| Merge base with `main` | `2f8bf1d9c3bbf3cf7beb3504d508854155272c3e` |
| Last known good public line | `main` at merge-base above (pre-NES merge) |
| Implementation PR | [#9](https://github.com/ferxalbs/mate-x/pull/9) — **do not merge** without founder authorization |

## If PR #9 has not been merged

1. Do **not** merge the PR.
2. Do **not** tag `v0.1.2` or create a GitHub Release.
3. Leave draft PR open or close with comment linking this document.
4. Continue development on a new branch from `main` if abandoning NES for this cut.

## If PR #9 was merged to main

1. Create a revert PR that reverts the merge commit (or the contiguous NES stack) using `git revert -m 1 <merge_sha>`.
2. Do **not** force-push `main`.
3. Re-run `bun run verify` on the revert branch.
4. If a public tag/release already exists, publish a superseding patch only after a new candidate; mark the broken release notes as yanked in CHANGELOG.

## Application / data rollback (installed macOS)

1. Quit MaTE X.
2. Remove `/Applications/MaTE X.app` (or the install location used).
3. Optional: remove app support data under the user’s Application Support directory for MaTE X if a full reset is required (destroys local EngineeringTask DB and settings).
4. Reinstall last known good build (v0.1.1 or prior RC) if available.
5. Workspace `.mate-x/` evidence remains in repositories; it does not automatically unlock Git without the app’s GitGate.

## Artifact discard

Internal candidate artifacts live under:

```text
artifacts/rc/v0.1.2-rc.2/
out/make/
```

These are **gitignored** and may be deleted safely. Discarding them does not roll back git history.

## Database / migration notes

- v0.1.1 → EngineeringTask migration is idempotent.
- Rolling back the app version does not delete migrated rows automatically.
- Prefer restoring from a user-data backup if one was taken before migration dogfood.

## Communication

If a bad candidate was distributed internally:

1. State the SHA and artifact hashes that must not be used.
2. Point to the superseding candidate when available.
3. Never claim notarization/Gatekeeper success for unsigned builds.
